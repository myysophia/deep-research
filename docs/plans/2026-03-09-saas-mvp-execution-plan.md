- Auth 模块以 `NIN-47` 作为 Epic，子任务单独拆分并挂依赖。
- 每个 issue 必填：

  - Estimate
  - Dependencies

  - 风险说明

## 4. 风险清单与缓解

- 风险：登录态与后端会话不一致导致频繁 401。
  - 缓解：客户端会话刷新与服务端鉴权统一使用 Supabase 规范。
- 风险：密码重置链接失效和邮件延迟影响转化。
  - 缓解：提供重发入口与失效态兜底文案。
- 风险：支付/配额主线抢占 M1 资源。
  - 缓解：将登录页闭环列入 M1 出口门槛，未完成不得进入 M2。

## 6. 首期 MVP 收敛范围（2026-03-09 晚间补充）

首期上线目标从“完整收费 SaaS”临时收敛为单一闭环：

- 用户可邮箱注册
- 用户可邮箱登录
- 登录后无需填写 API Key / 访问密码即可进入产品并开始使用

### 当前确认的 MVP 缺口

1. 注册流程不能依赖 Supabase 控制台中的“邮件确认开关”，否则会阻断“注册后立刻可登录”。
2. 代理调用原先依赖 `ACCESS_PASSWORD`，与“必须先登录”目标不一致。
3. 前端默认仍偏向 BYOK，本地持久化设置可能覆盖 SaaS 默认配置。
4. 设置页暴露 `mode` / `accessPassword` 等字段，会误导首期 SaaS 用户。

### 本轮落地策略

- 注册改为服务端 `admin.createUser + email_confirm=true`，保证新用户可立即登录。
- SaaS 默认通道收敛为：`proxy + pollinations + openai-fast`。
- 客户端 proxy 调用优先携带 Supabase access token；旧 `ACCESS_PASSWORD` 仅作为兼容回退。
- 设置页在 SaaS MVP 模式下隐藏 `mode`、`provider` 与 `accessPassword` 主路径。

### 对应 Linear 任务

- `NIN-64` Auth-Server 接入 Supabase Service Role 与统一会话校验
- `NIN-65` SaaS-Proxy 将 AI/Search 代理鉴权切换为 Supabase 登录态
- `NIN-66` SaaS-Onboarding 登录后默认进入平台代理配置并收敛设置项
- `NIN-67` MVP-QA 邮箱注册登录即用闭环验收
